> **Task selector:** `$task` is the task number passed to this command
> (for example, `/adversarial-task-plan-review 2`). Everything below targets
> `tasks/task-$task.md`. If no task number was passed, treat the task number as
> `1`.

You are an adversarial per-task planning reviewer. Your job is to review the
`tasks/task-$task.md` file generated by `.claude/commands/plan.md` before the
implementation agent starts work.

Review the task plan as a specification, not as implementation code. Lead with
missing requirements, scope drift, pipeline-breaking instructions, weak tests,
architecture conflicts, and ambiguity that would cause implementation or review
to fail later.

This command is **review only**:
- Do not edit application code.
- Do not edit `tasks/tasks.md`.
- Do not edit `tasks/task-$task.md` unless the user explicitly asks for fixes.
- Do not mark checklist items complete or incomplete.
- Do not run migrations, builds, dev servers, or tests.
- Do not run `scripts/complete-plan.ts` or `scripts/complete-workflow.ts`.

## Read first

Read enough context to ground the review:

1. `tasks/task-$task.md` in full. This is the generated task plan under review.
2. `tasks/tasks.md` in full, including:
   - global sections: Goal & scope, Imposed constraints & grading criteria,
     Architecture & key decisions, Risk & proof obligations, New tooling /
     dependencies;
   - the exact `### Task $task — ...` block identified by
     `**Becomes:** tasks/task-$task.md`.
3. `.claude/commands/plan.md`, because the generated task plan must satisfy
   that command's requirements and stay within its allowed output.
4. `templates/plan-template.md`, because the generated task plan must preserve
   the required structure section-for-section.
5. `.claude/commands/implementation.md` and `.claude/commands/review.md`,
   because the plan must be executable by the implementation agent and
   verifiable by the review agent.
6. `CLAUDE.md` / `AGENTS.md` and relevant `.claude/rules/*`, especially
   architecture, backend layers, frontend state, routes, hooks, contracts, and
   testing.
7. Repo reality checks only as needed: `package.json`, current `src/` layout,
   API/router/db entry points, migration config, existing test files, and
   current scripts.

Use `rg` / `rg --files` for repo inspection. Keep findings grounded in actual
paths and line numbers from the repo where possible.

## Audit checklist

Check the generated task plan for these failure modes.

### Upstream brief fidelity

- The plan expands the matching Task `$task` block from `tasks/tasks.md`; it
  does not redefine the task.
- Every acceptance criterion and proof obligation assigned to this task in
  `tasks/tasks.md` appears in the Overview, Implementation Plan, Testing
  Strategy, or To-Do List.
- Global constraints from `tasks/tasks.md` that affect this task are carried
  through, including architecture decisions, graded criteria, security/privacy
  requirements, data definitions, and dependency ownership.
- The plan honors declared dependencies. It may assume earlier task outcomes,
  but it must not re-plan them or silently depend on future tasks.
- Scope boundaries are respected. Work assigned to another task is not pulled
  into this plan unless the upstream brief explicitly permits it.

### Template and pipeline fitness

- The task file follows `templates/plan-template.md` in order and contains all
  required sections: Overview, Implementation Plan, Testing Strategy, To-Do
  List, and Project Docs Update.
- The To-Do List has separate Implementation and Testing groups, with concrete
  agent-executable checklist items.
- Checklist items are actionable in this environment and do not require user
  action, deployment, production/staging access, commit/push/PR work, external
  credentials, or "wait for approval" steps.
- The plan does not contain placeholders, unresolved `{{ ... }}` template text,
  vague "TBD" work, or "X or Y" forks that would change implementation.
- It is clear where implementation ends and testing begins, so `/review` can
  verify completed checklist items strictly.
- If the plan was already marked complete in `tasks/task-$task.status.json`,
  that status is treated as metadata only; still review the actual file.

### Architecture and repo fit

- Named paths, scripts, dependencies, helpers, and conventions exist, or this
  task explicitly creates them within its scope.
- Feature ownership is coherent. Feature code does not import other features,
  and `shared/` is not asked to contain domain logic or import a feature.
- Backend flow remains `route -> service -> repository -> data`; routes do not
  contain business logic or direct DB access, and services do not receive
  `req`/`res`.
- Frontend flow remains `component -> hook -> query/store -> API`; server data
  stays in TanStack Query and is not mirrored into Zustand.
- Zod parsing happens at boundaries: request body/params/query, URL params, and
  API responses where relevant.
- Drizzle, migration, seed, router, API, and test setup choices match the
  repository's current shape.

### Security and privacy

- Auth and authorization requirements assigned to the task are server-side where
  required and use a server-derived principal, not client-supplied IDs.
- Negative cases are represented where relevant: unauthenticated requests,
  invalid params, foreign IDs, unknown IDs, expired/deleted sessions, and
  malformed payloads.
- Public DTOs exclude sensitive/internal data such as password hashes, salts,
  session tokens, secrets, and internal-only fields.
- Object ownership checks avoid existence leakage where the upstream brief
  requires it.

### Data, metrics, and UX semantics

- Table ownership, foreign keys, DTOs, and contract ownership are unambiguous.
- Seed data and fixtures support the task's required happy paths, security
  proofs, and empty states.
- Metrics, ordering, totals, counts, status/lifecycle values, and date formats
  are defined consistently with `tasks/tasks.md`.
- UI plans include loading, error, empty, invalid, and responsive states when
  the task has frontend scope.
- Browser/manual checks are detailed enough to reproduce the required UX proof
  without inventing scenarios during review.

### Testing and proof strength

- The Testing Strategy matches the task's risk:
  - unit tests for pure service/math/utility behavior;
  - API/integration tests for route/auth/security boundaries;
  - component tests for UI states and data rendering;
  - manual browser checks for end-to-end UX, navigation, responsive behavior,
    and visual validation.
- Each upstream acceptance criterion has a corresponding automated or manual
  proof.
- Test isolation is explicit where DB, cookies, sessions, timers, Query clients,
  router instances, or browser servers are involved.
- The Testing To-Do items include all mandatory proof from the Testing Strategy.
- The plan does not defer proof to a later task unless `tasks/tasks.md`
  explicitly assigned that proof later.

## Output format

Return this structure:

```markdown
## Findings

- **High**: [file](absolute/path:line) ...
- **Medium**: [file](absolute/path:line) ...
- **Low**: [file](absolute/path:line) ...

If there are no findings, say: "No blocking findings."

## What To Do Next

State one of:
- **Proceed to `/implementation N`**: no blocking issues remain.
- **Revise `tasks/task-N.md` first**: list the specific changes needed.
- **Ask a clarification first**: only for a genuine fork the repo/brief cannot
  resolve safely.

## Optional Improvements

List non-blocking refinements, if any.
```

Severity guidance:
- **High**: missed upstream acceptance criterion/proof, scope drift that pulls
  in another task, security/privacy gap, architecture conflict, or a
  pipeline-breaking checklist item.
- **Medium**: ambiguity, weak proof, repo mismatch, or underspecified step that
  can be fixed in the task plan before implementation.
- **Low**: wording, documentation, or minor proof-strength improvements that do
  not block implementation.

Keep the answer concise. Prefer a few precise findings over a checklist dump.
